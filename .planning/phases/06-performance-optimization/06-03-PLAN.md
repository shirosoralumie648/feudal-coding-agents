<objective>
Promote existing local scanners into a live execution-path guardrail.

Purpose: The code and sensitive-info scanners already exist, but they are library utilities rather than a guardrail. This plan scans executor-produced artifacts before verifier acceptance. High risk output fails closed, while lower risk findings remain visible as diagnostics.

Output: A deterministic execution scanner helper, orchestration integration before verifier dispatch, and focused tests proving blocking, redaction, and non-blocking diagnostics.
</objective>

1. Import `scanCodeSecurity` and `shouldBlockExecution` from `./code-scanner`.

3. Define:
   `export interface ExecutionArtifactInput { id?: string; kind?: string; name?: string; content: unknown }`

10. Redact sensitive contexts before adding diagnostics by calling `redactSensitiveInfo(context, sensitiveMatches)`.

    - When `report.blocked === true`, the flow transitions with `execution.failed`
    - When blocked, the flow appends an `execution-report` artifact containing a `securityScan` object
    - When blocked, the flow appends the executor run summary before persisting
    - When blocked, the flow never calls `options.runStep` for `phase: "verification"`
    - When `report.blocked === false`, existing verifier flow still runs
    - `apps/control-plane/src/services/orchestrator-flows.test.ts` covers blocked and non-blocked paths

  <action>
Update `apps/control-plane/src/services/orchestrator-flows.ts`.

Concrete behavior:
1. After executor `executeStep` succeeds and `executorRun = executeStep.run`, compute:
   `const securityScan = scanExecutionArtifacts(executorRun.artifacts);`
2. If `securityScan.blocked`:
   - create `const failedTask = transitionTask(task, { type: "execution.failed" });`
   - create a report content object:
     `{ result: "blocked", reason: "execution_security_scan_blocked", securityScan }`
   - append this as `execution-report` with `appendArtifact(failedTask, executorRun.id, "execution-report", reportContent)`
   - append executor run with phase `"execution"` using `appendRun(...)`
   - return `options.persistTask(taskWithReportAndRun, "task.execution_security_blocked")`
3. Do not dispatch verifier in the blocked branch.
4. If not blocked, continue with the existing `execution.completed` and verifier flow.

Create `apps/control-plane/src/services/orchestrator-flows.test.ts` if it does not exist.
