    ===== Factory Function =====

    ```typescript
    import type { RunStep, AwaitStep, StepResult } from "./orchestrator-runtime";
    import type { WorkflowTemplate, TemplateStep, TemplateInstantiation, TemplateCondition } from "./workflow-template-types";
    import { interpolateParams, validateParameters } from "./workflow-template-params";
    import type { TaskRecord } from "@feudal/contracts";

    export interface WorkflowTemplateEngine {
      executeTemplate(options: {
        template: WorkflowTemplate;
        instantiation: TemplateInstantiation;
        baseTask: TaskRecord;
        runStep: RunStep;
        awaitStep: AwaitStep;
        persistTask: (task: TaskRecord, eventType: string) => Promise<any>;
      }): Promise<any>;
      resolveExecutionOrder(steps: TemplateStep[]): string[];
      evaluateCondition(condition: TemplateCondition, stepOutputs: Record<string, unknown>): boolean;
    }
    ```

    ===== resolveExecutionOrder(steps) =====

    Returns an array of step IDs in dependency-respecting topological order (Kahn's algorithm):
    1. Build adjacency: for each step, record its `dependsOn` IDs as incoming edges
    2. Seed queue with steps that have empty `dependsOn` (no incoming edges)
    3. Process queue: for each dequeued step, remove it as a dependency from other steps; if any step now has no remaining unsatisfied dependencies, enqueue it
    4. If not all steps processed by the end, there is a cycle — throw `new Error(\`Circular dependency detected: \${cycleSteps.join(" -> ")}\`)`

    ===== evaluateCondition(condition, stepOutputs) =====

    Evaluates a single TemplateCondition against accumulated step outputs per D-04:
    1. Look up `stepOutputs[condition.sourceStepId]` — the raw output object from that step's run
    2. Navigate the `condition.path` (dot-separated, e.g., "result.status" means `output.result.status`)
    3. Apply `condition.operator`:
       - "equals": `actualValue === condition.value`
       - "notEquals": `actualValue !== condition.value`
       - "contains": `String(actualValue).includes(String(condition.value))`
    4. If the path does not exist in the output, return false (condition not satisfied)

    ===== executeTemplate(options) =====

    Main execution flow:
    1. Validate parameters: call `validateParameters(template.parameters, instantiation.parameters)`. If errors, throw `new Error(\`Parameter validation failed: \${errors.join(", ")}\`)`
    2. Resolve execution order: call `resolveExecutionOrder(template.steps)`
    3. Initialize `stepOutputs: Record<string, unknown>` to track outputs from completed steps
    4. For each step ID in execution order:
       a. Look up the `TemplateStep` by ID from `template.steps`
       b. Evaluate conditions: if `step.conditions` is non-empty, check each condition via `evaluateCondition(condition, stepOutputs)`. If ALL conditions evaluate to false, skip this step (continue to next)
       c. Interpolate step config: `const interpolatedConfig = interpolateParams(step.config, instantiation.parameters)`
       d. If `step.type === "approval"`:
          - Call `awaitStep(baseTask, { label: "approval-gate", prompt: interpolatedConfig?.prompt as string ?? "Approve?", actions: interpolatedConfig?.actions as string[] ?? ["approve", "reject"], metadata: { stepId: step.id } })`
          - Store run result: `stepOutputs[step.id] = result.run.artifacts[0]?.content`
          - Update baseTask from result.task
       e. Otherwise (non-approval step):
          - Call `runStep(baseTask, step.type, { agent: step.agent, messages: [{ role: "user", content: JSON.stringify(interpolatedConfig ?? {}) }], metadata: { stepId: step.id } })`
          - Store run result: `stepOutputs[step.id] = result.run.artifacts[0]?.content`
          - Update baseTask from result.task
    5. Persist the final task state via `persistTask` and return

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Template execution input | TemplateInstantiation (user-provided parameters) crosses into the engine — validated by validateParameters before interpolation |
| Step condition evaluation | Step outputs from agent runs cross into condition evaluator — paths are dot-navigated on untrusted objects; missing paths are handled safely |
| RunStep/AwaitStep boundary | Engine dispatches external calls to orchestrator runtime — engine never creates runs directly |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-05 | T (Tampering) | interpolateParams injection | mitigate | interpolateParams validates that every ${params.xxx} reference exists in the parameters record; missing references throw before reaching RunStep |
| T-03-06 | D (Denial of Service) | Dependency cycle detection | mitigate | resolveExecutionOrder detects cycles and throws before execution begins — infinite loops prevented at resolution time, not runtime |
| T-03-07 | E (Elevation of Privilege) | executeTemplate — agent dispatch | mitigate | Engine only calls RunStep/AwaitStep with the agent specified in the template step; it cannot escalate privilege beyond the template definition |
| T-03-08 | I (Information Disclosure) | Step output accumulation | accept | Step outputs from completed steps are passed to condition evaluation via stepOutputs; this is by-design behavior for D-04 condition chaining. No PII in agent outputs. |
| T-03-09 | E (Elevation of Privilege) | DEFAULT_TEMPLATE | mitigate | The default template is a hardcoded constant matching the existing orchestrator-flows.ts behavior — it cannot be modified at runtime and uses the same agent names as the current hardcoded flow |
</threat_model>
