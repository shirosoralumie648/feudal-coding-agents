# 三省六部制 Agent 集群架构设计

## 1. 设计目标
- 将复杂任务拆解为“决策-审议-执行”闭环，降低单 Agent 决策偏差。
- 用组织分权实现稳定扩展：新增能力时只扩展对应“部”而非重构全系统。
- 通过制度化审查和审计实现可控、安全、可追责的自动化协作。
- 同时支持战略任务（多阶段项目）与战术任务（即时请求）。

## 2. 核心架构体系（组织映射）

### 2.1 三省（决策与治理核心）

#### 1. 中书省：策略规划集群 (The Secretariat - Strategy Cluster)
**核心职责**：将模糊意图转化为精确的、经过事实核查的技术蓝图。

| 角色 (Role) | 职能 (Function) | 关键动作 (Action) | 产出 (Output) |
|---|---|---|---|
| **中书舍人 (Liaison)** | **需求沟通** | 追问用户细节、提供技术栈选项、确认边界 | 用户画像、PRD |
| **中书侍郎 (Analyst)** | **系统拆解** | 将业务需求转化为系统架构、任务分级 (P0/P1/P2) | 系统设计文档 (SDD) |
| **采风司 (Fact Checker)** | **事实核查** | **强制核查**第三方库版本、API 签名、依赖可行性 | 置信度报告、引用清单 |
| **专业顾问团** | **方案草拟** | 前端/后端/架构/安全/运维/测试顾问出具详细方案 | 分领域子方案 (Sub-Plan) |

#### 2. 门下省：审议监督层 (The Chancellery - Audit Cluster)
**核心职责**：系统质量的“守夜人”，负责风险拦截与质量把控。

| 角色 (Role) | 对标中书 | 审查模式 | 核心动作 |
|---|---|---|---|
| **给事中 (Auditor)** | 中书侍郎 | **逻辑一致性审查** | 检查 SRS 是否符合用户原意，维护**风险热力图** |
| **谏官团 (Critics)** | 顾问团 | **对抗性盲审** | 反对派谏官强制挑刺，独立发表意见 |
| **影子给事中** | - | **抽样后评估** | 对 P1 快速任务进行 10% 随机抽检，生成训练数据 |

#### 3. 尚书省：执行编排层 (The Department of State Affairs - Execution Cluster)
**核心职责**：资源调度中心，负责将“蓝图”转化为“工单”。

*   **尚书令 (Orchestrator)**：总调度官，维护全局状态机，处理任务死锁与级联熔断。
*   **军机处 (Fast-Track)**：直属特种部队，负责 P0 简单任务的极速执行（跳过三省流程）。

---

### 2.2 六部执行体系（自生长与高并行架构）

六部不再是静态的 Agent 集合，而是具备**“细胞分裂”**能力的自组织集群。

#### 1. 职能与执行矩阵 (Function-Execution Matrix)

| 六部 (Ministry) | 核心职能 | 下辖司 (Specialized Agents) | 关键机制 |
|---|---|---|---|
| **吏部 (Personnel)** | **资源调度** | **考功司**（能力画像）、**选部**（任务派发） | 动态匹配 Worker 能力，记录信用分 |
| **户部 (Revenue)** | **成本核算** | **度支司**（Token 预算）、**金部**（API 额度） | **减法激励**：删代码奖励 > 写代码奖励 |
| **礼部 (Rites)** | **协议治理** | **主客司**（API 契约）、**祠部**（Prompt 规范） | **Mock 优先**：生成契约即生成 Mock Server |
| **兵部 (War)** | **安全攻防** | **职方司**（静态扫描）、**武库司**（密钥托管） | **异步扫描**：不阻塞开发，但有权熔断 |
| **刑部 (Justice)** | **质量验收** | **都官司**（单元测试）、**比部司**（代码审计） | **测试舱壁**：每模块独立容器，批量冒烟 |
| **工部 (Works)** | **工程建设** | **营缮司**（全栈开发）、**屯田司**（UI 设计）、**虞衡司**（运维） | **工具绑定**：强制调用环境探测工具 |

#### 2. Worker 自生长机制 (Agent Mitosis)
当**尚书令**下发一个复杂的工程框架（如：一个包含 10 个页面的 React 项目）时，工部 Agent 会根据框架复杂度自动繁衍：
*   **工部-营缮司 (Master Dev)**：接收总框架，将其拆解为一个个独立的“原子组件”或“功能模块”。
*   **分裂 (Mitosis)**：Master Dev 根据模块数量，自动实例化出多个 **子 Worker (Sub-Workers)**：
    *   *示例*：Master 发现有 3 个复杂页面，它会瞬间克隆出 3 个 `Page-Worker`，分别负责各自页面的开发。
    *   *生命周期*：子 Worker 随任务创建，任务结束并合入主干后自动销毁，节省资源。

#### 3. 动态并行编排 (Dynamic Parallelism)
为了最大化吞吐量，六部执行遵循 **“能并尽并”** 原则，引入 **Mock 优先** 策略消除硬依赖：

| 部门 | 并行执行策略 (Parallel Strategy) | 汇聚点 (Sync Point) |
|---|---|---|
| **工部 (Works)** | **Mock-First 开发**：<br>1. 礼部定义 API 契约后，**立即生成 Mock Server**。<br>2. 前端连接 Mock 开发，后端连接契约开发，**完全解耦**。<br>3. 前端无需等待后端接口实现，实现 0 等待。 | **集成联调**：只有在上线前，才将 Mock 替换为真实后端服务。 |
| **刑部 (Justice)** | **批量冒烟 (Batch Smoke)**：<br>1. 引入 **测试舱壁 (Test Bulkhead)** 模式，每个原子模块完成后立即启动**隔离测试容器**。<br>2. 单模块失败不扩散，不阻塞其他模块测试。<br>3. 测试 Agent 与开发 Agent 1:1 结对，同步编写测试脚本。 | **增量验收**：模块级测试通过即视为“局部交付”，最终仅需做集成验收。 |
| **兵部 (War)** | 实时监听代码变更，**异步进行** 静态扫描与漏洞分析。 | **兵部-合规签章**：上线前的最后一道锁。 |
| **户部 (Revenue)** | 实时监控各子 Worker 的 Token 消耗，实现 **秒级成本看板**。 | **预算熔断**：一旦超支，立即叫停所有并行分支。 |

#### 4. 矩阵管理逻辑
*   **横向（尚书令）**：负责项目进度和部门间汇聚。
*   **纵向（各部长官）**：负责本部门内部的 Worker 繁衍、质量把控和知识积累。

### 2.3 权责边界与冲突处理
- 中书省不直接执行工具调用，只负责计划正确性。
- 门下省拥有“否决权”，但不改写业务内容，只给审议意见。
- 尚书省拥有“调度权”，但必须遵循门下省准入结论。
- 冲突优先级：安全（兵部）> 合规（刑部）> 预算（户部）> 进度（吏部/工部）。
- 任何高风险任务必须触发双审：门下省 + 刑部联合审批。

### 2.4 扩展职能层 (Extended Functional Layer)

#### 1. 宰相府：统一接入层 (Prime Minister's Office - Intake Layer)
**核心职责**：在任务进入三省六部前的“前置清洗”与“标准化”，减轻中书省负担。
*   **Intake Agent**：
    *   **标准化 (Normalization)**：将用户口语化的需求转化为标准 JSON 结构。
    *   **脱敏 (Sanitization)**：自动识别并掩盖敏感信息（如 AK/SK、个人隐私）。
    *   **去重与合并 (Deduplication)**：识别重复提单，合并相似请求（如 3 人同时报同一个 Bug）。

#### 2. 都水监：数据治理司 (Directorate of Waterways - Data Governance)
**核心职责**：专司数据完整性与状态一致性，填补工部/刑部对“数据变更”的监控盲区。
*   **核心动作**：
    *   **Schema 演进**：负责数据库迁移计划 (Migration Plan) 的生成与校验。
    *   **影子数据验证**：在沙箱环境中模拟数据回填与清洗，确保无损。
    *   **回滚脚本校验**：强制要求所有数据变更必须附带经过测试的逆向脚本 (Down Migration)。

#### 3. 鸿胪寺与理藩院：对外与边缘接口 (Court of Colonial Affairs & State Ceremonials)
*   **鸿胪寺 (External Interface)**：负责跨 Agent 网络的标准通信协议（如 FIPA），处理外部 Agent（如供应链 Bot）的身份认证与契约对齐。
*   **理藩院 (Edge Integration)**：管理“封疆大吏”（端侧 Agent）。将轻量级任务下放到设备本地 NPU 执行，仅将结构化意图上报“中央”。

#### 4. 大理寺：法律审计 (Grand Court of Revision - Legal Audit)
**核心职责**：专司“不可逾越的红线审查”，优先级高于兵部。
*   **核心动作**：版权合规检查（防止生成侵权代码）、PII 隐私数据泄露拦截、开源协议污染审计。

### 2.5 能力自生长与进化 (Capability Evolution & Self-Growth)
系统的能力不应是静态的，而应随任务执行而自我增强。

*   **吏部-考功司（动态反馈）**：基于《后评估报告》自动优化 Agent 的 System Prompt。例如发现某 Agent 常忘写 Loading 状态，自动在其 Prompt 中注入规则。
*   **工部-军器监（私有工具制造）**：当发现某类重复性工作极多时，自动编写 Python 脚本并注册为新 Tool，实现从“使用工具”到“创造工具”的飞跃。

## 3. 运行机制与流程（动态视角）

### 3.1 标准执行全流程（闭环）
1. 用户请求进入接入层，生成 TaskSpec。
2. 中书省产出执行草案（子任务拆解、候选六部分工、预算估算）。
3. 门下省进行审议：
   - 通过：进入尚书省编排。
   - 退回：带原因返回中书省重写计划。
   - 拒绝：触发人工介入或终止。
4. 尚书省按依赖关系并行/串行调度六部 Agent。
5. 刑部与兵部在执行中做实时合规与安全拦截。
6. 工部回填工具调用结果，户部更新成本记账。
7. 门下省执行结果复核，通过后归档，输出可追溯报告。

### 3.2 需求澄清与方案确立（三省交互闭环）
为了解决“用户一句话需求”与“系统需要详尽方案”之间的鸿沟，引入**“双重审查 + 御批”**机制。

#### 1. 流程图解
`用户` <-> `中书舍人` (沟通) -> `中书侍郎` (分析) -> `门下给事中` (系统审查) -> `中书顾问团` (详细设计) -> `门下谏官团` (技术审查) -> `用户` (御批) -> `尚书省` (执行)

#### 2. 详细步骤
1.  **用户下旨（Initial Prompt）**：用户输入简短需求（如：“做个贪吃蛇”）。
2.  **中书省拟旨（Drafting & Clarification）**：
    *   **舍人沟通**：中书舍人与用户多轮对话，确认业务边界（Web版、单机、像素风）。
    *   **侍郎分析**：中书侍郎将业务需求转化为《系统需求说明书 (SRS)》，包含功能列表、非功能需求。
3.  **门下省一审（System Review）**：
    *   **给事中审计**：检查 SRS 是否准确反映用户意图？逻辑是否自洽？
    *   *驳回*：SRS 存在歧义 -> 打回中书侍郎重新分析。
    *   *通过*：进入详细设计阶段。
4.  **中书省详设（Detailed Design）**：
    *   **顾问团分工**：前端顾问出 UI 草图，后端顾问出 API 接口，运维顾问出部署方案。
    *   **侍郎整合**：汇总为《详细设计文档 (DDD)》。
5.  **门下省二审（Technical Review）**：
    *   **谏官团审计**：前端谏官查 UI 交互，安全谏官查权限漏洞。
    *   *驳回*：接口定义不规范 -> 打回对应顾问修改。
    *   *通过*：生成《决策简报》。
6.  **用户御批（Final Approval）**：
    *   用户查看简报（含成本、风险、核心设计）。
    *   **准奏 (Approve)**：方案锁定，转交尚书省执行。
    *   **再议 (Reject)**：用户不满意方案 -> **回滚至步骤 2**，中书舍人重新询问需求。
7.  **尚书省执行（Execution）**：只有获得“准奏”状态的任务，才会被分发给六部执行。

#### 3. 决策简报（DecisionBrief）模板
这是呈送给用户的最终确认单，必须控制在 **一屏以内**，且支持**交互式决策**。

```markdown
# 决策简报：贪吃蛇游戏开发

## 1. 方案对比 (Solution Comparison)
| 方案 | 技术路线 | 耗时 | 成本 | 风险 | 适用场景 |
|---|---|---|---|---|---|
| **A. 保守方案 (推荐)** | HTML5 + Canvas + Vanilla JS | 3 mins | $0.5 | 低 | 追求稳定、轻量级 |
| **B. 激进方案** | React + Three.js (3D效果) | 8 mins | $1.2 | 中 | 追求视觉效果、炫技 |

## 2. 核心影响 (Impact Analysis)
- **模块影响**：新增 `SnakeGame` 组件，不影响现有布局。
- **数据变更**：新增 `localStorage` 键值 `snake_high_score`。
- **服务停机**：否（热更新）。

## 3. 待确认风险
- ! **注意**：本地存储在清理浏览器缓存后会丢失最高分数据。

## 4. 决策参数 (Decision Parameters)
- [x] 允许 Mock 交付 (先上线 UI，后端接口暂用假数据)
- [ ] 允许预算上浮 20% 以换取更高代码质量

[ 准奏方案 A ]   [ 准奏方案 B ]   [ 修改 (Modify) ]
```

### 3.3 示例：一次“新功能上线”任务如何流转
- 中书省：拆解为“需求分析、代码实现、测试验证、发布回归”。
- 门下省：校验是否满足安全与合规基线，必要时要求补充测试门禁。
- 尚书省：调度工部执行 CI/CD，刑部审查变更记录，户部评估资源成本。
- 执行完成后门下省复核，通过即归档并输出上线总结。

## 4. 关键支撑机制（核心设计）

### 4.1 统一数据与通信协议（解决信息孤岛）

#### 1. 核心数据模型：《分布式皇榜（Distributed Edict）》
为了解决 PRD、SRS、DDD 格式混乱的问题，同时**打破中心化阻塞**，定义基于 **“快照+分支合并 (Snapshot + PR Merge)”** 的分布式对象模型。

**防御性设计：逻辑锚点 (Logic Anchors)**
在 JSON 中植入**不可压缩的断言**，防止长链路传递中的逻辑漂移。
```json
{
  "edict_id": "UUID-v4",
  "logic_anchors": [
    {
      "id": "LA-001",
      "assertion": "禁止在浏览器端暴露 Secret Key",
      "severity": "CRITICAL",
      "verification": "REGEX_MATCH" // 下游必须用正则校验，不能只靠理解
    }
  ],
  // ... 其他字段
}
```

**去中心化特性：**
*   **PR 模式协作**：皇榜由中书侍郎作为 Owner 维护主干，其他顾问通过 Pull Request 提交方案片段。
*   **乐观锁 (Optimistic Locking)**：顾问先本地生成方案，提交时检测冲突（类似 Git merge conflict）。
*   **分片读写 (Sharding)**：前端顾问只锁定 `specifications.frontend` 分片，后端顾问锁定 `specifications.backend` 分片，互不阻塞，并行写入。

```json
{
  "edict_id": "UUID-v4",
  // 向量时钟，仅用于审计追溯，不用于实时冲突解决
  "vector_clock": { ... },
  
  // 分布式分片（各部门并行操作的区域）
  "shards": {
    "frontend_shard": { 
      "owner": "front_consultant", 
      "status": "DRAFTING", 
      "content": { ... } 
    },
    "backend_shard": { 
      "owner": "back_consultant", 
      "status": "REVIEWING", 
      "content": { ... } 
    },
    "test_shard": { 
      "owner": "test_consultant", 
      "status": "WAITING", 
      "content": { ... } 
    }
  },
  
  // 依赖关系图（定义并行边界）
  "dag_graph": {
    "frontend_design": ["system_analysis"],
    "backend_design": ["system_analysis"],
    "api_contract": ["frontend_design", "backend_design"], // 只有API需要等待前后端汇合
    "test_plan": ["api_contract"]
  }
}
```

#### 2. 临时高频通信：《军机处黑板 (Grand Council Blackboard)》
为了解决《皇榜》在高并发讨论时的锁竞争问题，引入**黑板模式**。
*   **机制**：设立共享的临时内存区，各顾问可在此发布“便签”和“高亮指针”。
*   **用途**：用于头脑风暴和中间态信息的快速交换，仅在达成最终决议时才 Commit 到正式的《皇榜》中。

#### 3. 通信总线：《驿站（The Courier System）》
解决“二元反馈”与“信息滞后”问题，支持 **异步并行流**。
*   **GitLab CI 式审议**：
    *   **预审非阻塞**：中书侍郎提交 SRS 后，门下给事中的审查与专业顾问的详细设计并行进行。
    *   **批注而非驳回**：门下省在行内批注建议（Comment），中书省选择性采纳，而非全盘驳回。
    *   **风险分级**：高风险点（如安全、核心架构）强制修改，低风险（如注释风格）标记 TODO 后续处理。
*   **DAG 驱动的事件流**：不再是线性的“中书->门下->尚书”，而是基于 DAG（有向无环图）的动态触发。
    *   *示例*：一旦中书侍郎发布了 `SRS`，系统自动**同时**唤醒前端顾问、后端顾问、安全顾问。他们**并行工作**，互不等待。
    *   *汇聚点*：只有在“接口对齐”这个节点，才需要前后端顾问坐下来开个小会（Merge State），之后又继续并行。
*   **增量修正机制**：
    *   支持 **`Patch Proposal`**：门下省发现小问题，可直接发布补丁，中书省确认即可。
    *   支持 **`Real-time Commenting`**：在 JSON 分片上进行批注。

#### 3. 集成保障机制：《演武场（The Drill Ground）》
针对“模块能跑，组装就挂”的集成难题，引入**契约测试**与**虚拟沙箱**机制，确保所有积木块的凹凸槽完美匹配。

*   **契约先行 (Contract-First Design)**：
    在写任何代码前，必须先在《皇榜》中锁定 **API 契约（OpenAPI/Swagger）**。
    *   **Mock Server 自动生成**：一旦契约锁定，工部自动生成 Mock 服务。
    *   **并行开发**：
        *   前端 Agent 对着 Mock 接口开发，无需等待后端写完。
        *   后端 Agent 对着契约写实现，必须通过契约校验。
    *   **硬性约束**：任何不符合契约的请求/响应，会被中间件直接拦截，强迫双方对齐。

*   **持续集成沙箱 (CI Sandbox)**：
    每个 Agent 完成模块（如一个函数、一个组件）后，**不能直接提交**，必须先进入沙箱“演练”。
    *   **单元测试 (Unit Test)**：刑部-都官司负责，保证模块内部逻辑正确。
    *   **组件集成测试 (Component Integration Test)**：将新模块放入一个**最小化运行环境**（Docker 容器），与上下游的 Mock 对象进行交互。
    *   **全链路冒烟 (E2E Smoke Test)**：在沙箱中模拟真实用户请求，跑通“注册-登录-下单”主流程。
        *   *失败处理*：如果冒烟失败，代码被**自动退回**给对应的工部司局，严禁合入主分支。

*   **依赖熔断与降级**：
    *   如果前端 Agent 发现后端某个接口迟迟未完成，可以申请**“临时豁免”**，使用 Mock 数据继续开发，但在《皇榜》中标记为 `PENDING_INTEGRATION`，确保上线前必须补齐。

### 4.2 弹性与应急机制（解决重、慢、死锁问题）

针对“系统过重、响应过慢、死循环”等痛点，引入**变法机制**。

#### 1. 快速通道与自适应路由：《军机处（The Grand Council）》
引入 **“可计算分层治理”** 机制，通过**任务复杂度评分卡**量化任务风险，自动选择执行路径与治理深度，拒绝“一刀切”与“凭感觉”。

**任务复杂度评分卡 (Task Complexity Scorecard)**

| 维度 (Dimension) | 权重 | 评分标准 (Scoring Criteria) |
|---|---|---|
| **代码范围** | 1x | 单文件=1, 多模块=3, 跨服务=5 |
| **依赖程度** | 1x | 无=0, 低(1-2)=1, 中(3-5)=2, 高(>5)=4 |
| **安全敏感度** | **熔断** | 无=0, 内部数据=2, **核心/支付/鉴权=5 (强制 P2)** |
| **需求模糊度** | 1x | 清晰=0, 部分模糊=2, 极其模糊=4 |
| **回滚难度** | 1x | 易(无状态)=0, 中(有数据变更)=2, 难(不可逆)=4 |

*注：若命中门下省“风险热力图”的高温模块，基础分自动 +2。*

**方向盘接管模式 (Steering Mode)**
在工部执行过程中，允许用户通过宰相府发送“急递铺”指令（Out-of-band Signal）。尚书令接收后触发**“执行流挂起”**，将当前状态打包，退回中书省进行**“增量重规划”**，而非从头再来。

**分级路由策略**

| 级别 (Level) | 判定阈值 (Threshold) | 路由路径 (Routing Path) | 治理深度 (Governance Depth) |
|---|---|---|---|
| **L1 微任务 (P0)** | **0 ~ 4 分** | **军机处 -> 工部 -> 交付**<br>*(跳过中书/门下/尚书)* | **轻型治理**：<br>1. 无需 `ImperialEdict`，仅用简单 `TaskSpec`。<br>2. 取消门下省审议，保留工部+刑部的自动化测试。 |
| **L2 标准任务 (P1)** | **5 ~ 9 分** | **中书 -> 尚书 -> 工部 -> 交付**<br>*(跳过门下省一审)* | **标准治理**：<br>1. 使用 `ImperialEdict Lite`（仅保留 logic_anchors 和 dag_graph）。<br>2. 门下省一审合并到尚书省调度前的快速检查。 |
| **L3 战略任务 (P2)** | **≥ 10 分** 或 **安全=5** | **完整三省六部流程** | **重型治理**：<br>1. 全量文档流程。<br>2. 引入**“流式审议 (Streaming Review)”**：中书侍郎每完成一个子系统分析，立即推送给门下给事中并行审议。 |

**反升级机制 (Governance Deflation)**
*   **自动降级**：若某类任务（按特征指纹聚类）连续 **N 次**（如 5 次）在 P1/P2 流程中**未触发任何驳回且回滚率为 0**，系统自动下调其“基础复杂度系数”（如将其安全敏感度评分 -1），使其更容易落入低治理层级。
*   **目的**：防止体制膨胀，确保系统越用越轻。

**动态降级 (Dynamic Downgrade)**
*   如果一个 P2 任务被中书省拆解为 10 个子任务，其中 8 个是 P0 级别的（如写 8 个独立的 UI 组件），这 8 个子任务会自动降级走军机处通道，只有剩下的 2 个核心任务走完整流程。

**架构健康度监控**
为了防止架构“虚胖”，引入核心监控指标：
*   **协调开销比 (Coordination Overhead Ratio, COR)**
    $$ COR = \frac{\text{Token(中书+门下+尚书+吏+户+礼)}}{\text{Token(工部+兵部+刑部)}} $$
*   **熔断阈值**：若 $COR > 30\%$，说明管理层话太多，干活人太少。系统自动触发**“精简模式”**，强制提升 P0/P1 任务的判定阈值，减少审批层级。

#### 2. 死锁仲裁：《御史台（The Censorate）》
解决“中书省一直被门下省驳回”或“代码一直过不了测试”的死循环（Infinite Loop）。

*   **三振出局法 (Three-Strike Rule)**：
    *   如果同一个任务在同一环节被驳回 **3次**。
    *   自动触发 **御史台介入**，暂停自动化流程。
    *   **御史大夫 (Arbitrator Agent)** 分析死锁原因：
        *   如果是“需求不清”，强制回滚到用户沟通环节。
        *   如果是“技术无法实现”，生成《无能为力奏折》反馈用户，请求降级或取消。
        *   如果是“Agent 幻觉”，尝试更换执行 Agent（如从 GPT-4 降级到 Claude-3 或反之）。

#### 3. 故障保护机制 (Failure Protection)
在弹性机制基础上，补充**“级联熔断”**与**“智能降级”**策略。

*   **级联熔断 (Cascading Circuit Breaker)**：
    *   **工部级联**：如果工部（开发）连续 **3次** 无法通过刑部（测试）验收，不仅触发御史台介入，还应**暂停尚书省所有同类任务的调度**，防止垃圾代码堆积。
    *   **上下文风暴保护**：当 `ImperialEdict` 压缩后仍超过 16k tokens，触发任务**强制拆分**（如将一个大页面拆为两个子组件任务），拒绝单一大任务执行，防止 OOM。

*   **熔断协调器 (Circuit Breaker Orchestrator)**：
    解决多重防御机制（户部预算、刑部测试、兵部安全）相互触发导致的**“死锁与过度惩罚”**问题。
    *   **状态仲裁**：当触发多个熔断条件时（如：重写导致预算超支），协调器会判断**主次责任**。如果是为了修复 Bug 而导致的预算轻微超支（<20%），协调器会授予**“临时豁免权”**，抑制户部熔断，优先保证任务闭环。
    *   **僵死检测**：监控任务状态流转，若发现任务在“熔断-恢复-熔断”循环中超过 2 轮，强制触发御史台人工介入。

*   **多模型智能降级 (Model Fallback)**：
    *   **Agent 降级策略**：当 GPT-4 级 Agent 连续失败（如生成代码语法错误），自动降级到 Claude-3.5 或 Gemini 尝试（利用模型多样性解决幻觉死角）。

*   **人机协作边界 (Human-in-the-loop Boundary)**：
    引入**“渐进式人工介入”**，避免过度依赖人工导致的流程卡顿。
    *   **Level 1 异步通知**：预算微量超支、低风险幻觉。系统仅记录日志并异步通知用户，**不阻塞**任务执行。
    *   **Level 2 同步等待**：预算显著超支（>50%）、需求漂移。系统暂停并等待用户输入，但设定超时阈值（如 10分钟）。超时后自动降级为“记录+继续执行”或“挂起”。
    *   **Level 3 强制阻断**：高危操作（如兵部标记的删库、改权限）。**必须**人工显式确认，否则永久阻塞该指令。

#### 4. 知识管理与上下文优化：《翰林院 (The Imperial Academy)》
解决“文档过长撑爆 Context Window”与“史馆检索成本高”的问题，并引入**知识可信分层**机制，防止陈旧与错误信息污染决策。

*   **分层摘要机制**：
    *   **对上（给用户/尚书令）**：只看 `DecisionBrief`（几百 tokens）。
    *   **对平级（给其他顾问）**：只看 `InterfaceSpec`（接口定义），不看具体实现代码。
    *   **对下（给 Worker）**：只给 `TaskSpec` 和必要的 `Context Fragment`，不给全量项目文档。

*   **史馆三维知识标签 (Three-Dimensional Knowledge Tagging)**：
    为每条知识（如 Case、Rule、Code Snippet）打上三层元数据，确保检索的精准性与安全性。
    1.  **可信度标签 (Trust Level)**：
        ```json
        { "source_type": "official_doc | internal_case | agent_summary | human_note", "verified": true, "confidence": 0.92 }
        ```
    2.  **时效标签 (Time Validity)**：
        ```json
        { "valid_from": "2025-01-01", "valid_to": "2025-06-30", "version_scope": { "react": "^19.0.0", "node": ">=20" } }
        ```
    3.  **适用域标签 (Applicability)**：
        ```json
        { "domain": ["frontend", "auth", "deployment"], "scenario": ["greenfield", "refactor", "bugfix"] }
        ```

*   **知识召回白名单 (Recall Whitelist)**：
    针对**高风险任务（L3/P2）**，实施严格的知识过滤策略，优先只召回：
    1.  **官方文档**（source_type = official_doc）
    2.  **已验证内部最佳实践**（verified = true）
    3.  **最近 90 天内成功案例**（freshness <= 90d）
    *   *目的*：彻底屏蔽未验证的历史摘要与陈旧代码，降低幻觉被“历史垃圾”放大的风险。

*   **史馆冷启动与遗忘机制**：
    *   **向量化摘要 (Vectorized Summary)**：将历史任务产出转化为可检索的 Embedding，而非每次召回完整 JSON。Agent 仅在向量匹配度高时才调取完整档案。
    *   **遗忘曲线 (Forgetting Curve)**：按任务热度衰减史馆数据权重。三个月前未被引用的“支付模块”失败经验，其检索优先级应显著低于昨天的类似错误，防止陈旧知识干扰当前决策。

### 4.3 质量防御体系 (Quality Defense System)

#### 1. 门下省主动治理 (Proactive Governance)
门下省不仅是被动“盖章”，更应是系统质量的“守夜人”。

*   **影子审议 (Shadow Review)**：
    针对走“P1 快速通道”跳过一审的任务，实施 **10% 风险加权抽样 (Risk-Weighted Sampling)**。
    *   **热力图驱动**：基于风险热力图调整抽检率。高温模块（如支付、鉴权）抽检率提升至 50%，低温模块（如静态文案）降低至 5%，保持总体 10% 的吞吐平衡。
    *   **非阻塞执行**：影子审议在后台默默进行，不影响任务的正常上线。
    *   **训练反馈循环**：
        *   如果影子给事中发现“这个 P1 任务其实有隐患”，它不会叫停任务，而是生成一份《后评估报告》。
        *   这份报告会被喂给 **中书省** 作为反例教材（Few-Shot Learning），提升中书侍郎下次拟策的质量。

*   **风险热力图 (Risk Heatmap)**：
    门下给事中实时维护一张全局热力图，识别系统的“薄弱地带”。
    *   **模块维度**：如果“支付模块”频繁被影子审议发现问题，该模块会自动升温，下次针对该模块的任务会被**强制降级**走 P2 完整流程。
    *   **Agent 维度**：如果某个 Worker Agent 的代码回滚率偏高，会被标记为“高风险”，限制其接单权限。

#### 2. 刑部测试左移与影子模式 (Shift-Left & Shadow Mode)
为了在代码产生前就拦截 Bug，刑部将防线前移。

*   **静态契约测试 (Static Contract Test)**：
    在工部开始写第一行代码**之前**，刑部-都官司先对《详细设计文档 (DDD)》中的 Swagger/OpenAPI 定义进行静态扫描。
    *   **规范性检查**：检查 API 是否符合 RESTful 标准，参数命名是否驼峰统一。
    *   **安全预判**：识别潜在的 SQL 注入风险点（如：参数直接拼接 SQL），强制要求使用预编译语句，否则不许开工。

*   **影子模式回归 (Shadow Regression)**：
    针对 **P1/P2** 级重要任务，启用**“双轨运行”**机制。
    *   **沙箱并行**：新代码不直接替换旧版本，而是在沙箱中与旧版本并行运行。
    *   **流量回放**：将生产环境的真实请求（脱敏后）同时发给新旧两个版本。
    *   **差异比对 (Diff Check)**：
        *   比对输出结果是否一致（对于重构类任务）。
        *   比对性能指标（耗时、内存占用）。
        *   **准入标准**：仅当输出一致率 > 99% 且新功能测试通过时，才允许正式交付。

#### 3. 事实核查机制：《采风司（The Fact-Checking Bureau）》
为了根治 LLM 的“一本正经胡说八道”（如编造 API、依赖不存在的库），在中书省增设 **两阶段采风策略**。

*   **预采风 (L1 - Pre-Fact Check)**：
    在**中书侍郎分析阶段**进行，核查显性依赖（用户提到的技术栈、明确指定的库）。
*   **深度采风 (L2 - Deep Fact Check)**：
    在**顾问团设计阶段**并行进行。当顾问发现新依赖或需要确认 API 细节时，立即触发快速核查（5秒内返回置信度），而非阻塞整个设计流程。

*   **引经据典 (Citation Check)**：
    *   **依赖库核查**：如果方案说要用 `pip install fast-gpt-v5`，采风司会联网去 PyPI/NPM 查是否存在该版本。
    *   **API 签名核查**：如果方案调用了 `OpenAI.create_cat()`，采风司会去查官方文档看有没有这个方法。
*   **置信度报告 (Confidence Report)**：
    *   生成一份报告：`{ "dependency_valid": true, "api_valid": false, "hallucination_score": 0.8 }`。
    *   如果**幻觉分 > 0.1**，直接打回中书侍郎，门下省根本不会看一眼。
*   **治理三板斧**：
    *   **Schema 强制引用**：所有代码生成必须基于已验证的 Schema（如 OpenAPI Spec），禁止“自由发挥”。
    *   **经济惩罚**：如果一个 Agent 编造了事实被采风司抓包，扣除其 Token 预算额度（降低其优先级）。
    *   **审计问责**：所有幻觉记录记入“史馆”，作为 Agent 能力画像的负面权重。
    *   **史馆可信度权重 (Archive Reliability)**：防止冷启动期的误报污染。前 100 个任务采用人工验证史馆记录，确认无误后才设为永久规则，避免将“真实但小众的 API”误判为反模式。

#### 4. 深层缺陷防御矩阵 (Deep Defense Matrix)
针对 LLM 的认知、行为、系统性缺陷，构建多维防御体系。

*   **对抗性审议 (Adversarial Review)**：
    解决**“从众偏误 (Conformity Bias)”**。
    *   **异构审查强制 (Heterogeneous Review)**：在门下省（审计）和兵部/刑部（质量）中，**强制要求**使用与中书/工部不同的异构基座模型（如中书用 GPT-4，门下用 Claude-3.5）。利用模型潜空间盲区的差异，防范“模型同质化塌陷”导致的盲目认同。
    *   **异议强制 (Dissent Mandate)**：门下省谏官团中设置 **“反对派谏官 (Devil's Advocate)”**，其 KPI 是**必须找出至少 1 个实质性缺陷**才能通过，禁止全票通过。
    *   **盲审机制 (Blind Review)**：专业谏官在审查时，**看不到**其他谏官的意见，避免锚定效应。

*   **工具强制绑定 (Tool Binding)**：
    解决**“工具惰性 (Tool Laziness)”**。
    *   **Schema 强制**：某些字段（如 `api_latency`）必须由工具填充，禁止 LLM 手写。
    *   **执行锁**：工部 Agent 在生成代码前，**必须调用环境探测工具**获取当前依赖版本。
    *   **工具健康度旁路 (Tool Health Bypass)**：解决工具失效导致的死锁。当某工具连续失败率 > 20%，降级为“声明模式”。工部 Agent 需在代码中标记 `[TOOL_UNAVAILABLE: 待验证]`，兵部在静态扫描时对此类标记给予更高风险权重。

*   **减法文化 (Subtraction Culture)**：
    解决**“叠加偏好 (Addition Bias)”**。
    *   **净增代码量 (Net Code Change) 考核**：防止“先写后删”的套利行为。只奖励**最终交付版本**相对于**初始基准**的负净增量（Negative Net Change）。如果中间过程产生大量增删（Churn）但净增为正，不予奖励。
    *   **强制重置**：当工部代码连续 2 次测试失败，**强制要求重写 (Reset)** 而非修补 (Patch)。

*   **安全与认知防御**：
    *   **跨层防火墙**：每层 Agent 接收输入时，必须通过**提示词注入检测**，防止恶意指令级联。
    *   **认知谦逊协议**：Agent 必须显式声明置信度（如“我对 React 19 的了解截止 2024 年”），超出边界的知识需由**采风司**验证。

### 4.4 异常处理与事务一致性 (Exception Handling & Consistency)
针对“任务非成功即失败”的二元论局限，引入更工程化的 **Saga 事务模式** 与 **部分交付模型**。

#### 1. Saga 补偿事务模式 (Saga Compensation Pattern)
为每个执行步骤定义“正向操作”与“逆向补偿”，确保系统处于一致状态。

*   **步骤定义结构**：
    ```yaml
    step: deploy_service
    action: 
      - call_deployment_api
    success_criteria: 
      - health_check_pass == true
      - error_rate < 1%
    compensation_action: 
      - rollback_deployment
      - restore_previous_config
      - notify_audit_ledger
    ```
*   **执行逻辑**：
    *   当步骤 N 失败时，尚书省自动逆序执行步骤 N-1, N-2 ... 的 `compensation_action`，实现**最终一致性回滚**。

#### 2. 部分交付模型 (Partial Delivery Model)
打破“全有或全无”，允许在风险可控的前提下进行柔性交付。

*   **人工辅助分级交付 (Human-Guided Partial Delivery)**：在 `PARTIAL_SUCCESS` 状态下，系统生成交互式 Web IDE 界面，允许用户在《决策简报》中直接修改高亮的出错代码并批准通过，而非全盘驳回。

| 状态 (Status) | 定义 | 典型场景 | 后续动作 |
|---|---|---|---|
| **SUCCESS** | 100% 预期达成 | 完美上线 | 归档 |
| **PARTIAL_SUCCESS** | 核心功能可用，非核心失败 | 5个模块完成4个，仅“用户头像”上传失败 | 降级上线 + 生成补救工单 |
| **WAIVED_WITH_RISK** | 带病上线（经人工批准） | 性能略低于预期但需紧急修复线上 Bug | 上线 + 监控 + 24h内优化 |
| **SOFT_FAIL** | 预检查失败，未产生副作用 | 依赖库版本不兼容，未开始写代码 | 驳回重试 |
| **HARD_FAIL** | 执行失败且产生副作用 | 数据库迁移一半失败 | 触发 Saga 回滚 |
| **ROLLED_BACK** | 已完成回滚，环境已恢复 | 部署失败后自动回滚成功 | 任务终止，分析原因 |

### 4.5 发布与灰度治理 (Release & Feature Flags)

#### 1. 统一发布门禁对象 (Unified Release Gate)
定义标准化的 JSON 对象，作为上线前的“通关文牒”，取代散落在各部门的口头确认。

```json
{
  "release_gate": {
    "tests_passed": true,          // 刑部：单元测试 + 冒烟通过
    "security_signed": true,       // 兵部：无高危漏洞
    "data_migration_verified": true, // 都水监：回滚脚本已验证
    "budget_within_limit": true,   // 户部：未超支
    "rollback_plan_ready": true,   // 尚书省：回滚流水线就绪
    "observability_ready": true    // 工部：监控埋点已上线
  }
}
```

#### 2. 特征开关与灰度体系 (Feature Flags & Canary)
将“渐进式交付”写入架构原则，降低多 Agent 自动执行的“爆炸半径”。

*   **默认开关 (Default Off)**：所有新功能代码必须包裹在 Feature Flag 中，默认为关闭状态。
*   **灰度策略 (Canary Strategy)**：
    *   **Phase 1**：仅对内部测试用户 (White List) 开放。
    *   **Phase 2**：按 1% -> 5% -> 20% -> 100% 逐步放量。
    *   **自动熔断**：若 Error Rate > 1%，自动回滚开关至 Off 状态。

## 5. 技术架构与部署（落地视角）

### 5.1 分层技术架构
1. **接入层**：API Gateway、消息总线、鉴权网关。
2. **治理层**：中书省/门下省策略与审议引擎。
3. **编排层**：尚书省调度器、状态机、**动态 Worker 管理器**。
4. **执行层**：六部二十四司与**弹性 Worker 节点**。
5. **数据层**：分布式皇榜、翰林院知识库、审计仓。
6. **观测层**：指标、链路追踪、**成本看板**。

### 5.2 核心组件
- **统一任务模型（TaskSpec）**：目标、约束、预算、SLA、敏感级别。
- **审议规则引擎（Policy Engine）**：可配置准入规则与驳回理由模板。
- **编排状态机（Workflow FSM）**：Draft -> Review -> Approved -> Execute -> Verify -> Close。
- **事件总线（Event Bus）**：异步广播状态变化，支持补偿与回放。
- **审计账本（Audit Ledger）**：记录每次决策依据、执行动作与工具调用。

### 5.3 部署建议（生产可落地）
- **控制面与执行面分离**：
  - **控制面**：中书省、门下省、尚书省（低频高价值决策）。
  - **执行面**：六部 Agent Worker 池（高并发可水平扩展）。
- **微服务化**：每部独立服务与独立扩缩容策略，避免单点瓶颈。
- **可靠性保障**：采用消息队列 + 幂等任务键，保障重试安全。
- **数据高可用**：关键存储三副本（配置中心、审计仓、任务状态仓）。
- **网络安全**：对外暴露统一入口，对内使用服务网格做零信任通信。
- **部署拓扑落地文档**：`docs/deployment-topology-control-execution-plane.md`

### 5.4 工程实施最佳实践 (Engineering Best Practices)

#### 1. 工作流引擎选型
*   **推荐方案**：采用 **Temporal.io** 或 **AWS Step Functions** 作为尚书令的底层引擎。
*   **理由**：原生支持 Saga 模式、状态机持久化、长时休眠（Sleep）以及宕机自动恢复，避免手写 Python 死循环导致的脆弱性。

#### 2. Agent 间通信协议
*   **强制结构化**：放弃纯文本拼接，强制采用 **Structured Output** (如 OpenAI JSON Schema) 进行部门间通信。
*   **协议标准**：参考 FIPA 或现代 Agent Protocol，确保“方言”统一。

#### 3. 可观测性增强：Token 拓扑图
*   **上下文漂移监控**：构建可视化工具，追踪每个环节输入的 Token 流向，监控关键需求（Anchors）在传递过程中的 Attention 衰减情况，辅助排查幻觉根因。

### 6.2 评测基准体系 (Evaluation Benchmarks)
为了验证三省六部制相对于单体 Agent 的优越性，并防止制度“形式主义化”，建立**“离线+在线”双重评测框架**。

#### 1. 离线基准评测 (Offline Benchmarking)
构建包含 7 类典型任务的标准测试集（Golden Dataset），对比 **单 Agent** vs **双 Agent (Planner+Executor)** vs **三省六部集群** 的表现。

*   **测试集任务**：
    1.  PRD 生成 SRS
    2.  API 契约设计
    3.  前后端协同开发
    4.  安全漏洞修复
    5.  代码小型重构
    6.  Bug 修复
    7.  多步骤复杂上线

*   **核心指标**：
    | 指标 (Metric) | 定义 | 目标 (Goal) |
    |---|---|---|
    | **一次通过率 (Pass@1)** | 首次提交即通过所有测试的比例 | > 40% (单 Agent 通常 < 10%) |
    | **幻觉率 (Hallucination Rate)** | 引用不存在的库/API 的次数 | < 1% (采风司拦截后) |
    | **Token 成本** | 完成同一任务消耗的 Token 总量 | < 2.5x 单 Agent (用成本换质量) |
    | **致命错误率** | 发生删库、泄密等高危操作的概率 | **0%** (兵部拦截) |

#### 2. 在线 A/B 实验 (Online A/B Testing)
在生产环境中对真实流量进行抽样实验，验证不同治理深度的实际效果。

*   **分组策略**：
    *   **实验组 A (10%)**：强制走 **L1 轻量治理**（军机处直通）。
    *   **实验组 B (10%)**：强制走 **L2 标准治理**（三省六部）。
    *   **对照组 (80%)**：按评分卡自动路由。

*   **观测维度**：
    *   **交付速度**：从提单到上线的平均时长。
    *   **回滚率**：上线后 24h 内发生回滚的比例。
    *   **用户修改率**：用户在“御批”阶段修改方案的次数（反映中书省满意度）。
    *   **安全告警率**：上线代码被安全扫描出的漏洞数。

#### 3. 组织级效能复盘 (Organizational Retrospective)
量化各部门的“治理有效性”，识别并裁撤低效环节。

| 部门 | 关键指标 (Key Metric) | 含义 | 阈值 (Alert Threshold) |
|---|---|---|---|
| **门下省** | **拦截有效率** | 被拦截的问题中，经确认确实存在风险的比例。 | < 30% (说明门下省在瞎指挥，需优化规则) |
| **中书省** | **方案采纳率** | 用户直接“准奏”无需修改的比例。 | < 60% (说明中书省不懂用户) |
| **工部** | **一次提交通过率** | 代码提交给刑部测试，一次性通过的比例。 | < 50% (说明工部能力不足或 TaskSpec 不清) |
| **刑部** | **误杀率** | 测试失败但实际上代码没问题的比例（如测试用例写错了）。 | > 10% (说明刑部在阻碍生产) |
| **兵部** | **漏报率** | 上线后被外部发现的安全漏洞数量。 | > 0 (零容忍) |
| **御史台** | **闭环成功率** | 介入死锁后，成功推动任务完成而非取消的比例。 | < 80% |

### 6.3 MVP 边界与非目标 (Non-Goals)
为了确保系统能够落地，必须明确第一版（Phase 1）的**受控裁剪策略**，避免过度设计。

*   **不做真正的多模型切换**：仅预留 LLM 调用接口，Phase 1 统一使用单一高性能模型（如 GPT-4），暂不引入模型路由矩阵。
*   **不做自动 Worker 分裂 (No Mitosis)**：工部 Agent 数量静态配置（如固定 3 个并发槽位），不做基于任务负载的动态繁殖。
*   **不做复杂向量知识库**：翰林院仅维护基于关键词检索的结构化 JSON 案例库，暂不引入 Vector DB 和 Embedding 检索。
*   **不做影子流量回放**：刑部测试仅限于 CI 沙箱内的合成数据测试，不接入生产环境流量。
*   **不做分布式分片写入**：`ImperialEdict` 采用中心化读写锁模式，暂不支持多顾问同时 PR Merge 的复杂并发。

### 6.4 演进路线图 (Roadmap)

#### Phase 1（2~4周）
- 建立三省主流程（计划、审议、编排）与统一 TaskSpec。
- 实现六部中的吏部、工部、刑部最小集。
- 打通审计链路与基础监控。

#### Phase 2（4~8周）
- 引入户部成本治理与兵部安全策略。
- 强化门下省规则引擎，支持可视化策略配置。
- 增加失败补偿、自动回滚、重试分级。

#### Phase 3（8~12周）
- 完整六部能力上线，支持跨域任务并行协同。
- 建立经验知识库与任务模板市场。
- 形成稳定的生产 SLA 与组织运营机制。

---

该架构的关键不是“多 Agent 并行”，而是“制度化分权 + 可追责闭环”。三省保证决策质量，六部保证执行专业化，二者通过规则和审计连接，最终形成可持续演进的 Agent 组织体系。
